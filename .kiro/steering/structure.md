# プロジェクト構造

## ルートディレクトリ
```
├── articles/           # Markdown形式の技術記事
├── books/             # 本のコンテンツ（将来的な利用予定）
├── images/            # 記事・本で使用する画像ファイル
├── .kiro/             # Kiro AI アシスタント設定
├── .vscode/           # VS Code ワークスペース設定
├── package.json       # Node.js プロジェクト設定
└── package-lock.json  # 依存関係ロックファイル
```

## コンテンツの整理

### Articlesディレクトリ (`/articles/`)
- すべての技術記事をMarkdownファイルとして格納
- 各記事は独立した `.md` ファイル
- ファイル名は説明的でケバブケース形式
- gitでディレクトリを維持するため `.keep` ファイルを含める

### Booksディレクトリ (`/books/`)
- 将来の本コンテンツ用に予約
- 現在は `.keep` ファイルのみ
- 本機能実装時に構造を決定予定

### Imagesディレクトリ (`/images/`)
- 記事や本で使用する画像ファイルを格納
- GitHub連携により自動的にZennにアップロード
- 記事ごとにサブディレクトリを作成して整理
- 例: `/images/記事スラッグ名/` 形式で整理

## ファイル命名規則
- **記事**: 説明的なケバブケース形式のファイル名
  - 例: `synology-nas-no-space-left-btrfs-metadata.md`
- **ディレクトリ**: 小文字、説明的な名前
- **画像ファイル**: 記事スラッグに対応するサブディレクトリ内に配置
  - 例: `/images/kiro-zenn-article-improvement-workflow/screenshot-1.png`
- **Keepファイル**: gitで空ディレクトリを保持するため `.keep` を使用

## コンテンツガイドライン
- すべてのMarkdownファイルには適切なYAML frontmatterを含める
- 視覚的な魅力のため適切な絵文字アイコンを使用
- 発見しやすさのため関連トピックでタグ付け
- 日本語と英語の両方のコンテンツをサポート
- 適切なシンタックスハイライトでコードブロックを含める

## 画像管理ルール

### 画像ファイルの配置
- リポジトリ直下の `/images` ディレクトリに配置
- 記事ごとにサブディレクトリを作成して整理
- 構造例:
```
images/
├── 記事スラッグ名1/
│   ├── screenshot-1.png
│   └── diagram.jpg
└── 記事スラッグ名2/
    └── example.webp
```

### 画像ファイルの制限
- **ファイルサイズ**: 3MB以内
- **対応拡張子**: `.png` `.jpg` `.jpeg` `.gif` `.webp`
- 制限違反時はデプロイエラーが発生

### 画像の参照方法
- 記事内では `/images/` から始まる絶対パスで指定
- 相対パスは使用不可
```markdown
# 正しい指定方法
![](/images/記事スラッグ名/screenshot-1.png)

# 誤った指定方法（使用禁止）
![](../images/記事スラッグ名/screenshot-1.png)
```

### 画像のライフサイクル
- GitHubリポジトリへのプッシュで自動アップロード
- リポジトリから削除するとZenn上からも削除
- 画像差し替え時は最大1分程度のキャッシュ遅延あり
